Java Secrets Notes

Sometimes when programming there are things that are done automatically, either for convenience or for necessity.

For Example:  When you construct a String

String str = "Hello World!";

Java recognizes that you're creating a String an secretly calls the constructor

String str = new String("Hello World!");

In order to fully understand objects and inheritance, there are a few of Java's Secrets that you need to know.

1)  The Default Constructor

All objects need a constructor.  If no constructor is provided by the programmer, Java will secretly create one automatically.

For example:  Consider the following classes

public class A
{
   public A()
   {
      System.out.println("Constructing A");
   }
}

public class B extends A
{

}

If you write a method that constructs an A object:

A a = new A();

You'll notice that "Constructing A" is printed out.  This can be expected because that's the only thing that A's constructor does.

But notice that B was not provided any constructor.  So you would think that nothing would happen if you did the following:

B b = new B();

If you run that snippet, you'll notice that "Constructing A" is also printed out.  This is very confusing because there is no constructor for B, yet somehow it's calling the constructor for A.  Keep in mind that constructors are NOT inherited directly in the same way that methods are.

What is happening:

Whenever there is no constructor specifically provided for a class, Java will secretly insert one.  This is known as the Default Constructor. 

The Default Constructor is always a zero parameter constructor with one line of code:

What is written by the programmer:

public class B extends A
{

}		 What Java does secretly:

public class B extends A
{
   public B()
   {
      super();
   }
}

 This means that when we construct a B object, A's constructor is called.

super() may not exist

The automatic default constructor is only provided if no other constructor is specifically provided.  If another constructor has be defined, the default one 'disappears'.

Consider if class A and B were the following.  Note that A's constructor has changed to take in an int as a parameter.  This means that A has no default constructor; the only constructor that you can call is one with an int.

public class A
{
   public A(int num)
   {
      System.out.println(“Constructing A with a ”+num);
   }
}

public class B extends A
{
   
}

The two classes above will not compile.  Like before, Java attempts to create a default constructor for B. 

public B()
{
   super();
}

The problem is that this time A's constructor takes in an int.  Since B extends A, there is no super().  The only super that B could call would need to have an int as a parameter.  This is not what the default constructor does.  The only option for a default constructor is a no-parameter constructor that calls the no-parameter constructor of the super class.

It's worth noting that if A and B both had no constructors defined, they would both get a default constructor automatically, which would compile fine.  B's call to super() would be supported by the fact that A was automatically given a default.  Adding public A(int num) ruined that.

2)  The Secret Super

public class A
{    
   public A()
   {
      System.out.println("Constructing A");
   }
}


public class B extends A
{
    public B(int num)
    {
        System.out.println("Constructing B with a "+num);
    }
}

If you compile and run the following construction statemet:

B b = new B(5);

You will get an unexpected result:

Constructing A
Constructing B with a 5

Again, A's constructor was called before B's constructor ran.

Secret Super!

We've already learned that when a constructor is not specifically designed, a default one is automatically created by Java.  The same is true for a call to super's constructor.  If you never call any super constructor, java will automatically call the default super constructor.

What is written by the programmer:

public class B extends A
{
   public B(int num)
   {
      System.out.println...
   }
}		 What Java does secretly:

public class B extends A
{
   public B(int num)
   {
      super();
      System.out.println..
   }
}

Notice that even though B's constructor is one that takes in a parameter, the automatic super call is always to the no-parameter constructor.  Every constructor that you've ever written either has a call to super, or one is provided automatically.  You only truly need a call to super's constructor if you want to call one that has a parameter.

3)  Secret Extension

Consider the following class:

public class C
{
    
}

Again, since no constructors are provided, Java provides a secret one:

public class C
{
   public C()
   {
      super();
   }
}

But wait!  How can super's constructor be called if C isn't extending anything?

It secretly does.  Every single object that you've created is secretly a subclass of a greater class named Object

What is written by the programmer:

public class C
{

}		 What Java does secretly:

public class C extends Object
{
   public C()
   {
      super();
   }
}

The Object class is at the very top of every inheritance hierarchy.  It is known as the COSMIC SUPERCLASS (Mr. Sacco is not making this up).

Read through the methods that an Object has. Don’t take any of them too seriously, many of them don’t work the way that you expect them to. Most of the time, they are only useful if they are overridden. The equals method, for example, doesn’t work the way you want it to. A good equals method is one that is overridden at the class’ level, such as String’s equals method that actually compares the letters of the String with one another.

Object's API

The only method of Object that you should know about is the toString() method. Every object inherits a toString method that will return a really ugly String. It starts with the name of the class, followed by an @ symbol and a number that is essentially the ID for the object.

For example, if we did not override Card’s toString, it would return a String like Card@10c9d49.

Polymorphism with Object:
Since every class extends Object. Polymorphism will work with it as well:

Object o1 = new Card(“A”,”S”);
Object o2 = “Hello World”;
Object o3 = new Wanderer(250,250);

The problem with this is that once any object is stored as an Object variable, it can only call methods that are defined in Object.  This is as limited as a variable can get.

4)  Secret Method Call:

We just learned that every single object has a toString method, even if you don’t provide one. This leads into our next secret.
:
If you recall, concatenation is when you join Strings together to create a larger String. With non-objects like ints and doubles the process is easy, but with objects there is not always an obvious way to arrange the instance fields and represent the object as a String. This is the whole purpose of the toString.

For example, assume there is a class named Taco and the following code happened:

Taco t = new Taco();
String greeting = “Hello and welcome to SaccoBell. Would you like to try our “+ t;

This is where the secret method call comes into play. An object is not a String, so Java secretly calls its toString method to aid in concatenation.

Taco t = new Taco();
String greeting = “Hello and welcome to SaccoBell. Would you like to try our “+ t.toString();

This will happen every time you need an object to be represented as a String.
