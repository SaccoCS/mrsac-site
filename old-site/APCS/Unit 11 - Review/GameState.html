Part C: Going Beyond - Implementing the GameState interface

The issue with using the GameState interface is that it is a challenge to test without an object that implements the GameState interface. This is now your task.

I will describe a GuessingGame class below, but if you would like to think of your own two player text-based game and get it running with your project, you may do that instead.

Create a class named GuessingGame, which implements the GameState interface. The way that a GuessingGame will work is that it will pick a random number from 1 to 20 and players will alternate taking turns guessing until one of them gets it right.

Its constructor should take in two Player objects to store as instance fields. Other good instance fields to have would be an String to represent the correct number, and an ArrayList<String> to represent the numbers from 1 to 20 (these are the 'moves' that the players can make). Lastly, you may want to have instance fields to represent the 'current' player (the one whose turn it is) and the 'winner' player. The winner variable would remain null until a winner was found, but the current variable should have one of the two player objects assigned to it.

Using those variables implement the rest of the GameState interface's methods. The makeMove method is the most work. It represents the current player making a move, represented by the parameter String which is the same as one of the Strings from the ArrayList. You will need to test if he/she is the winner. If not, the String should be removed from the ArrayList of numbers so that it isn't guessed again. Though it's cheating, have the toString return a String that represents the number to guess, along with the toString of the ArrayList of remaining numbers.

Finally we can test out our program. You will have to add a public static void main method to get everything started. Essentially you need a constructed GameDriver object and then to call its play method, but it's not that simple. Check the parameter for GameDriver's constructor to help you.

 

 

Part D (Optional): Going Beyond - The Human Player

As amazing as two completely random players is, human interaction is a little better. Create a class named HumanPlayer that is a subclass of Player. It can still take in a name as a parameter in the constructor, but the method that needs work is the getNextMove method. Add a Scanner statement that will help the user choose one of the options. You may either have the user type out the exact String he/she wants, or set it up so that all they have to do is type the 'number' of the move that they want to make.
