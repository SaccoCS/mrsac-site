Changing the Object with Mutator Methods:

So far we've been looking at objects as variables that hold information.  When the object is constructed the variables are set, and after that the only thing you can do is read the information through accessor methods.  This is not the whole truth.

The types of objects that we've been creating are immutable.  This means that they cannot be changed once the instance fields are set.  While some objects are immutable on purpose (i.e. String), but we are now going to shift our style to making objects where the instance fields can change.

 

Changable instance fields:

When we first started, our instance fields were public and we could access them directly.

Name n = new Name("Marty","McFly");
System.out.println(n);

n.first = "Steve";
n.last = "Irwin";
System.out.println(n);

This code would first print out "Marty McFly" and then "Steve Irwin"

Assuming that the instance fields are public, they can be accessed directly and modified however we want.  But ever since the instance fields were made private we have no way of changing the values.

 

Methods are not variables:

A common attempt at modifying instance fields is the following

Name n = new Name("Marty","McFly");
System.out.println(n);

n.getFirst() = "Steve";
n.getLast() = "Irwin";
System.out.println(n);

This attempt is on the right track, but it will not compile.  The getFirst an getLast methods don't actualy give you the instance field variables, they only copy the value.  This means that you cannot ever assign any value to a method.

 

Basic Mutator Methods:

In order to allow people to change the first or last name we are going to want the assistance of a method that's a little bit different than the accessors that we've been working with.

a) This method's purpose is to set the first variable to a specific value.  We don't know what that value is going to be right now, we will need to have a parameter

b) When we call this method, we expect the changes to happen inside the object, meaning that the method will not return anything.  Instead of a return type, we should put void.  void is the official return type for a method that doesn't return anything.  This means that the method will have no return statement.

 

The method should look like the following:

public void setFirst(String f)
{
   first = f;
}

Notice that nothing is returned.  The only effect of this method is that the first instance field has been changed to whatever the parameter was.  This is a very similar process to what the constructor is doing, it just happens at a different time.

 

What does the client/runner look like:

Name n = new Name("Marty","McFly");
System.out.println(n);

n.setFirst("Steve");
System.out.println(n);

 

Things to notice:

- instead of n.getFirst() = "Steve";, the setFirst method is called with "Steve" as the parameter to the method.

- No variable needed:  In previous method calls our goal was to get information out of the object.  This would always require a 'catching' variable .  Now our method's job has nothing to do with producing a value; all of its effects are inside the object and its return type is void.  This means a 'catching' variable would be useless.

String str = n.setFirst("Steve");

If you really think about the code above, what String would you even expect to be returned when you tell an object to change it's name?

 

Finishing up Name:

Add the setFirst method to your Name class, and add another method named setLast which should be for setting the last name.  Change your NameRunner to be the following:

public class NameRunner
{
   public static void main()
   {
      Name n = new Name("Marty","McFly");
      System.out.println(n);

      n.setFirst("Steve");
      n.setLast("Irwin");
      System.out.println(n);
   }
}

Run it to confirm that it prints out "Marty McFly" and "Steve Irwin".
