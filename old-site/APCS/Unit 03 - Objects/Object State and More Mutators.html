Object State and More Mutator Methods

Until recently we have been making objects that are purely for binding together a few pieces of information.  This is is mainly so that you can transfer around groups of related information in an organized manner.  For example, the Point class is little more than a way of encapsulating together an x and y. 

It's time to think of objects a little differently.

 

Objects that Change Over Time:

With the help of mutator methods we now have the ability to change the instance fields simply by calling a method.  This means that we can now create objects that change over time.

Consider the BankAccount object, which will represent a person's savings account.  It has instance fields to represent the Account Name and the balance of the account

public class BankAccount
{
   private double balance;
   private String accountName;
   
   public BankAccount(String name, double bal)
   {
      accountName = name;
      balance = bal;
   }
   
   public double getBalance()
   {
      return balance;
   }
   
   public String getAccountName()
   {
      return accountName;
   }
   
   public String toString()
   {
       return "Account Name: "+accountName+"\nBalance: $"+balance+"\n";
   }
   
   public void changeAccountName(String name)
   {
      accountName = name;
   }
}

By now we should be able to recognize and know how to use the constructor, basic accessors, and toString method.  The only mutator method is the changeAccountName method, which modifies the account name.

 

Why would we want to change the account's name?  Isn't it set in the constructor?

Yes, the initial balance and name are set by the constructor, but sometimes we need to think of an object as something that isn't permanent.  If the BankAccount object is to represent a "live" account that lasts for years, it may need the ability to change the name at some point.  In real life someone might want to change the name on their account if they get married or divorced. 

Copy the following Runner into your project. 

public class BankAccountRunner
{
    public static void main()
    {
        BankAccount b = new BankAccount("Sacco",25000);
        System.out.println("Initial Account:\n"+b);

        b.changeAccountName("Mister Sacco");
        System.out.println("After Name Change:\n"+b);
        
    }
}

When reading through the runner, think of it as a story over time.  For now all of our runners will run as fast as they can, but you can still imagine that they don't happen instantly.   In the method above, an extciting tale unfolds:

- Once upon a time, "Sacco" opened a bank account with $25,000
- After many years, he became "Mister Sacco" and decided to change his name

Fascinating!

The important thing to take away is that objects are defined by their instance fields.  Those instance fields can change over time.

 

State:

The state of the object is a general term that describes the current values stored in the instance field.  Since sometimes methods change the values in the instance fields, this means that they are changing the state of the object.

Consider the BankRunner above.  When you run it the two printlns print out two different states of the object.  In one state the name is "Sacco" and in the other state the name is "Mister Sacco".  This is the same object, but with a different state.

Initial Account:
Account Name: Sacco
Balance: $25000.0

After Name Change:
Account Name: Mister Sacco
Balance: $25000.0

The state of the account was changed when the name was changed.

 

Mutator Methods Change the State of an Object by Changing the Instance fields:

While accessor methods didn't have any affect on the state of the object, the main purpose of a mutator method is to affect the state.  The changeAccountName method's purpose was to change the account's name variable to a new value.

 

When designing a mutator method, always think about how it affects the state of the object:

Let's consider that we want to add a method to the BankAccount class that simulates depositing money in the account.  There are a few things that you want to think about when you're writing a mutator method:

1)  How should this affect the state of the BankAccount?  In this case we simply look at the instance fields and think logically at which one should be affected.  It should be clear that the balance variable is the one that would be affected by a deposit

2)  Parameters:  Do you need any extra information in order to perform the method's function?  In this case depositing will affect the balance variable, but by how much?  We could make it so that depositing always meant adding $50, but that wouldn't be a very useful method.  A good deposit method would allow the client to put any amount.  This means that the method should have a parameter

3) Should the method return anything?  Most of the time for mutator methods the answer is no, meaning the return type should be void.  When you think about the purpose of the method it's to change the state of the object, not to return any information; that's what accessor methods are for.

 

Once you've answered those three questions you can put the method together.  We know that the method should have a void return type and take in a single double as a a parameter.

public void deposit(double amount)
{
   balance += amount;
}

Remember that the goal is to change the state of the object by modifying the balance variable.  As you can see, that variable is increased by however much is in the parameter variable.

Add the following code to the end of the Runner's main method:

        b.deposit(4000);
       System.out.println("After Deposit:\n"+b);

It tests out the account's deposit method, passing in a 4000 as a parameter.  If you run the main method now, the last print out should read:

After Deposit:
Account Name: Mister Sacco
Balance: $29000.0

Notice that the balance has increased by 4,000 is now at 29,000.

 

Another Mutator Method: the penalty method

Banks love to penalize for things like overdrawing or going below a minimum balance.  The next method that we are going to add to the BankAccount class is going to be named deductPenalty.   It should simulate a $20 penalty being taken away from the balance of the account.

1) How should this affect the state?  It should subtract 20 from the balance

2) Do you need any extra information in order to perform the function? Nope.  We have access to the balance and the penalty is always 20

3) Does this method return anything? Nope.  Its purpose does not involve returning a value.

Now that we know the important information about the method we can write a method header with the appropriate retrun type and parameters and then write a code that will affect the state in the appropriate way.

public void deductPenalty()
{
   balance -= 20;
}

In this case, the state is affected by deducting 20 from the balance.
