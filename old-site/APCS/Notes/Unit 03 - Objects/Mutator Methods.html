Mutator Methods

 

Sometimes objects are designed to simply encapsulate together, such as the Point class or the Name class.  These objects mostly just hold the data that they are constructed with their entire lifetime, and their methods are mostly accessors.  This is not always the case.

Much of the time objects are meant to have data that changes as the object is being used.  The Fighter class, for example, is not meant to simply hold the initial health of the Fighter; it changes over time as the Fighter battles with another.

A Mutator method is one that changes (or mutates) the state of the object by changing the instance fields in some way. 

 

void

Since the purpose of a mutator method is to change the instance fields, they usually do not return any data.  When defining a method header, a return type must be provided.  When the word void is used as a return type, it signifies that no data will be returned from the method.

 

When calling a void method from a client class, don't confuse it with an accessor method and try to deal with a returned value.

For Example:  System.out.println is a void method, meaning the following would not compile:

String str = System.out.println("Hello World");

The variable str looks like it's trying to store a value, but by definition the println method is a void method and doesn't return anything.

 

Parameters:

Mutator methods often take in a parameter, but it is not always necessary.  It really depends on the situation.  If you were writing a method named setX, which is supposed to change the value of and instance field x, then you would probably need a parameter, but if you were writing a method named setXToZero, you can perform that task without any help from parameters.

 

Example:

Consider the BankAccount class below: 

public class BankAccount
{
    private double balance;
    private String accountName;

    public BankAccount(String name, double bal)
    {
        accountName = name;
        balance = bal;
    }

    public double getBalance()
    {
        return balance;
    }

    public String getAccountName()
    {
        return accountName;
    }

    public void setAccountName(String name)
    {
        accountName = name;
    }

    public void deposit(double cash)
    {
        balance += cash;
    }

    public void withdraw(double cash)
    {
        balance -= cash;
    }
    
    public void penalty()
    {
        balance -= 20;
    }

    public String toString()
    {
        return "Account Name: "+accountName+"\nBalance: $"+balance+"\n";
    }
}

BankAccount's Basic Mutator Methods

The method setAccountName is a Basic Mutator method.  The sole purpose of the method is to give the client a way of changing the account's name after it's been constructed.

BankAccount b = new BankAccount("Mr. Sacco",24000);
b.setAccountName("Mr. Amazing");

In the snippet of code above, the name and balance of the BankAccount object are intitially set to "Mr. Sacco" and 24,000.  After the setAccountName method is called, the name has been replaced with "Mr. Amazing".  This means that the object's state has been changed, or mutated.

 

BankAccount's Non-Basic Mutator Methods:

Not all mutator methods simply replace an instance field.  The withdraw and deposit methods are a different type.  They still take in an amount, but they do not simply replace the instance field.  Each of them is supposed to simulate a bank transaction, meaning they add or subtract from the balance.  When writing your mutator methods you should always make sure that you're a) changing at least one instance field, and b) changing it in an appropriate way, meaning not simply replacing the instance field with the new value.

 

Parameters are NOT ALWAYS REQUIRED

Don't get stuck in the mindset that all mutators need a parameter.  Since the bank's penalty is always 20, the penalty method doesn't need a parameter to tell it how much to deduct.  You only need parameters when the situation calls for it.
