int Division

When dealing with ints, much of the math works the way that we expect it to.  Consider the following snippet of code:

int i = 15;
System.out.println( i );
i = i + 5;
System.out.println( i );
i = i * 2;
System.out.println( i );
i = i - 3;
System.out.println( i );

The output for this would be:
15
20
40
37

 

The weird stuff occurs when dividing ints.  Consider the following:

int a = 35;
int b = 3;
System.out.println( a/b );

When we think about the output, we have to be careful.  The result of dividing two ints must be an int.  In real math, the answer would be 11.6666666666, but there is a rule that if you do math with two ints, the result must be an int.  Since ints can only hold whole numbers, the result of 35/3 is simply 11.

In mathematical terms, int division only results in the quotient

This means you must be very careful:

Consider the following code, which attempts to find the area of a triangle when given the base and the height:

String input = JOptionPane.showInputDialog("Please enter the base of the triangle");
int base = Integer.parseInt(input);

input = JOptionPane.showInputDialog("Please enter the height of the triangle");
int height = Integer.parseInt(input);

int area = 1/2*base*height;

Even though the formula looks right, doing it this way will guarantee a zero in the area variable.  This is because the first part of the formula that is calculated is the 1/2.  Since 1 and 2 are both ints, dividing them results in a 0.  From there it doesn't matter what you multiply by, because multiplying by the zero causes the result to always be 0.

It's worth noting that there is no perfect way to calculate the area of a triangle since even integer values for a base and height could give a non-integer area.  If we work the equation around a little bit, we can get closer than the guaranteed 0 that we previously were calculating.

int area = base*height/2;

Note that this time we are first doing the multiplication of the base and the height.  This should give us a number larger than 1 to divide, thus solving the issue from before.  It doesn't give you a perfect area because any decimals will be lost, but it's interesting to see how the order that you perform math in can change your result.
