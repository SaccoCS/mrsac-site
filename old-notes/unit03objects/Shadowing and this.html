Shadowing and this

When designing objects, a common issue occurs when using parameters.  Consider the semi-useless TwoIntObject class below, whose purpose is to store two ints:

public class TwoIntObject
{
   private int num1;
   private int num2;
   
   public TwoIntObject(int num1, int num2)
   {
      num1 = num1;
      num2 = num2;
   }

   public void setNum1(int n)
   {
      int num1 = n;
   }
}

This will compile, but it will not function the way that we want it to.  After the object is constructed, the instance field will never get the proper value.

 

Shadowing:

It is important to notice that there are two num variables: the instance field and the parameter.  This is known as shadowing the instance field, and it is completely legal.  The only issue is that duplicate variable names can cause confusion.  When you use duplicate names, the rule is that the non-instance field is the one used.  Look at the TwoIntObject class below again.  This time, the instance field is in red and the parameter is in blue.

 

public class TwoIntObject
{
   private int num1;
   private int num2;
   
   public TwoIntObject(int num1, int num2)
   {
      num1 = num1;
      num2 = num2;
   }

}

Notice that the constructor does NOTHING to the instance field.  On further inspection, all that the constructor does is replace the parameter with itself, which is not useful in any way. 

 

 

this

The word this is a special word in java that is known as the implicit paramter.  Whenever you use this is used as the variable, it means that it's talking about something that belongs to the object.  Now look at the TwoIntObject class again.  This time the constructor's parameter is assigned to this.num, which means that you are talking about the num variable that belongs to the object (the instance field).

public class TwoIntObject
{
   private int num1;
   private int num2;
   
   public TwoIntObject(int num1, int num2)
   {
      num1 = num1;
      num2 = num2;
   }
}

 

Using this allows us to distinguish between the parameter and the instance field, and this allows us to properly implement the constructor.

 

public class TwoIntObject
{
   private int num1;
   private int num2;
   
   public TwoIntObject(int num1, int num2)
   {
      this.num1 = num1;
      this.num2 = num2;
   }
}

It is not a great idea to shadow variables, but if you do you should make sure that you use this to specifically refer to the instance field.

 

 

 

 

Another way of shadowing occurs when you accidentally declare a local variable (non instance field) with the same name as the instance field.  Consider the setNum1 method that's been added to the TwoIntObject class

public class TwoIntObject
{
   private int num1;
   private int num2;
   
   public TwoIntObject(int num1, int num2)
   {
      this.num1 = num1;
      this.num2 = num2;
   }
   
   public void setNum1( int n )
   {
       int num1 = n;
   }
   
}

This time the parameter has its own unique name.  The issue is that instead of saying:

num1 = n;

it says:

int num1 = n;

This is a big mistake because adding the word int makes it a declaration statement that creates a brand new num1 variable.  This new variable is now shadowing the instance field, meanin that the real instance field remains unchanged (which is not the purpose of the method)
