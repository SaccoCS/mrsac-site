Mixing ints and doubles

One thing that we already know is that doubles can store ints, but ints can't store doubles.

double thisWorks = 5;
int notLegal = 4.9;

In the two lines above, the first line will compile fine, but the second line will have an issue because there is no way to represent 4.9 as an int.

 

Fear Division!!!

Previously we saw that int division is different than we're used to.  Since doubles can handle decimals, dividing two doubles ends up with a value that we are more used to seeing.

int num = 9/2;
double div = 9.0/2.0;

Both of the two lines above are performing a similar operation, but the top will give a value of 4 while the bottom gives a 4.5 .   Java looks at the two values being divided and chooses whether to do int division or double division.

 

What about this?

double div = 9/2;

There is a mixing of types here.  The variable is a double, but the division is between ints.  The order that Java processes through things is important.  Before Java even looks at the variable on the left it must first 'simplify' the right side of the equation.  9/2 has two ints, so the result is a 4.  This int division happens despite the fact that div is a double.  That comes into play after the int division occurs, when it's time to store that 4 Java will then recognize that div is a double and convert the 4 to a 4.0 before storing it.

 

Another tricky mix:

double result = 4/10 + 4.0/10.0;

In the code above, order of operations is very important.  If a person were to glance at the math above, he/she might think that since there is a double, all division will be double-style division.  This is incorrect.  Java actually looks at it using order of operations.  Let's look at it slowly:

 

4/10 + 4.0/10.0

The since division comes before addition, Java starts with the 4/10 on the left.  In that division there are only ints (despite the rest of the equation being full of them).  This means that 4/10 is a 0.

0 + 4.0/10.0

Now Java will move to the next division statement, the 4.0/10.0 .  In that statement, the division is between two doubles so double-style division occurs, evaluating to a 0.4

0 + 0.4

Finally the addition occurs.  Since there is a double mixed in, the result will be a double.

0.4

 

 

This is most important when you're dealing with formulas that use fractions.  For example:  The formula for a triangle's area is ½bh.

The most common way that people have problems is because they forget about int division:

double base = 14.9;
double height = 37.2;
double area = 1/2*base*height;

While this looks like it's following the rules of the formula, the result of this is always a 0.0 in the area variable.  Let's look at it slowly:

Since there is only multiplication and division in this expression, Java starts with the first division on the very left:

1/2*base*height

Looking at 1/2, one might think that the result is 0.5, but in reality int division is occurring.  1/2 results in a 0.

0*base*height

From here, Java will multiply 0 times the base, and then times the height and end up with a 0.0.

 

It's worth pointing out that putting the numbers in a different order will give a different result.

double base = 14.9;
double height = 37.2;
double area = base*height*1/2;

This time, since Java works left to right the base*height calculation happens first, resulting in the double 554.28

 

554.28*1/2

The next step is multiplying the 554.28 and 1, which results in 554.28 again:

554.28/2

Finally, the division occurs.  The difference this time is that a double is involved, so the division goes as expected and the correct result is obtained.

277.14
