if Statements

An if statement is used to make descisions within a program. Its basic structure is as follows:

if( /* boolean expression or value */ )
{
    /* statements to run if the boolean is true */
}

When run, the if statement will look at the boolean value passed into its parentheses. If it's true, the code within the curly braces will be run. If it's false, the curly braces are skipped all together.

For example, if you had an int named val and wanted to announce if it was positive, the if statement needs to check and see if the values assigned to val is greater than zero.

if( val > 0 )
{
   System.out.println( val + " is positive");
}

Java will look at the condition val > 0, which will evaluate to either true or false. In the case that it's true, the println will be run. If it's false, the println is skipped and nothing will happen.

if-else

In the situation above, the code was a little awkward. The if statement makes it so that you either get a println telling you that it's positive, or the program finishes without anything happenning. This is an oversimplified program, but let's fix it so that it always prints out that the number is either positive or negative.

To accomplish this, we're going to tack an else statement to the end of the if-statement.

if( val > 0 )
{
   System.out.println( val + " is positive");
}
else
{
   System.out.println( val + " is negative");
}

An else statement doesn't get a boolean condition of its own. It's always connected to an if statement. The else statement runs when the if that it's attached to doesn't. This means that when val > 0 evaluates to true, "____ is positive" will be printed, but if it's false, "____ is negative" will be printed.

There's No Way Past an if-else

Previously, our if statement either printed out some code, or it was skipped. Using an else statement attached to the if makes it so that it is impossible to get past our snippet of code without one of the two blocks of code running. It either runs one block or it runs the other, but never both.

The One Line Rule

The curly braces that follow an if statement allow us to have as many lines of code in each part of an if statement as we want, but sometimes we only need it to run exactly one line of code. If an if statement is only responsible for a single line of code, you are allowed to omit the braces. The example above may actually be written like so:

if( val > 0 )
   System.out.println( val + " is positive");
else
   System.out.println( val + " is negative");

This doesn't mean that it's necessarily wrong to write curly braces, and some students will always use them, but it is common to see simple if statements that don't use curly braces.

Using a boolean variable as the control for an if statement

The condition in the parentheses of an if statement doesn't have to be an expression, it simply has to result in a boolean value. If you have a boolean value assigned to a variable, you may simply put it alone into the if statement. Consider this code, which performs the same as the code above.

boolean isPos = val > 0;

if( isPos )
   System.out.println( val + " is positive");
else
   System.out.println( val + " is negative");

The first line will evaluate val > 0 and assign the resulting true or false value into the isPos variable. The if statement will then decide which statement to run based on the value of the isPos variable. The result is identical to the previous example.

Bad Style

Many students get stuck with the idea that you MUST use an expression of some sort, so they feel compelled to check to see if the boolean is true using an expression.

boolean isPos = val > 0;

if( isPos == true )
   System.out.println( val + " is positive");
else
   System.out.println( val + " is negative");

This is an unnecessary step. What the code above is doing is comparing the variable with a true value and generating another true value if they do. The way that this works, the result of the expression
isPos == true is identical to theisPos variable. This is just extra work and you will never see a true programmer doing this.
