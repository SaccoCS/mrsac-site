Scope

When discussing the scope of a variable, we are considering what parts of an object a variable is accessible from. The general rule is that a variable is accessible by any element that is within the same block of code (curly braces) as the variable's declaration statement.

Local Variables/Parameters Belong to the Method They're Declared In

Consider the AreaCalculators class, which is meant to have a collection of static method that will calculate the area of a geometric shape when given the dimensions.

public class AreaCalculators
{
   public static double triangleArea(double base, double height)
   {
      double area = base*height/2;
      return area;
   }

   public static double rectArea(double base, double height)
   {
      double area = base*height;
      return area;
   }
}

Notice that both methods have the same three variable names: base, height, and area. These are NOT the same variables. Each method get creates its own version of the variables that it declares. This means that it's perfectly ok for these methods to have the same name.

Nervous students who don't understand scope will often worry that they need different variables for every method. They end up having variables base1, base2, base3, base4, etc... While this is legal to do, it is unnecessary because each method is allowed to have its own version of a variable named base.

Instance Fields Are Shared by All the Methods of the Object

Instance fields aren't defined within any individual methods. This means that they are visible (and shared by) to every method in the object.

Consider the Name class below.

public class Name
{
   private String firstName;
   private String lastName;
   
   public Name(String first, String last)
   {
      firstName = first;
      lastName = last;
   }
      
   public String getFirstName()
   {
      return firstName;
   }

   public String getLastName()
   {
      return lastName;
   }
   
   public String getInitials()
   {
      return firstName.substring(0,1) + "." + lastName.substring(0,1)+".");
   }
}

Name's two instance fields belong to the entire object, meaning that every method is referring to the same two variables. If a method changes the variables, like the constructor does, then the value is changed for every method in the object.

Shadowing

Shadowing is a situation that occurs when you have the same exact name for a local variable that you do for an instance field. Consider the following simplified version of the Name class where the instance fields are green and the parameters are red.

public class Name
{
   private String firstName;
   private String lastName;
   
   public Name(String first, String last)
   {
      firstName = first;
      lastName = last;
   }
}

The way this is written, the instance fields have different names than the parameters of the constructor. This makes it easy to distinguish the difference between the two.

It is LEGAL to have instance fields and parameters with the same exact name. Just be careful.

While it might be counter intuitive, it is absolutely legal to give parameters the same name as instance fields. This causes some confusion though and you must be careful.

public class Name
{
   private String firstName;
   private String lastName;
   
   public Name(String firstName, String lastName)
   {
      firstName = firstName;
      lastName = lastName;
   }
}

The compiler will allow the above class to compile, but look at the blue variables inside the constructor. The constructor is within the scope of both the instance fields and its parameters. Now that they have identical names it's no longer clear which variables are the instance fields and which variables are the parameter. This constructor is now broken; you must be careful.

Rule: When shadowing, the LOCAL variable is chosen over the instance field by default

Whenever shadowing has occurred, when the shadowed variable is referred to, java sees the variable as the LOCAL version of variable. This means that Java views all the variables referenced inside the constructor as the parameters, and not the instance field.

public class Name
{
   private String firstName;
   private String lastName;
   
   public Name(String firstName, String lastName)
   {
      firstName = firstName;
      lastName = lastName;
   }
}

This causes a problem. The constructor needs to assign to the instance fields, but it shadowing has caused us to only assign back into the parameters. This causes the instance fields to always be null because they never recieve a real value.

Use this to refer specifically to the instance field

When put in front of a variable, the keyword this will tell Java that we are referring to the variable that belongs to the class,meaning the instance field. This is how to solve the problem above of not being able to reference the instance fields when shadowing. Consider the fixed Name class below.

public class Name
{
   private String firstName;
   private String lastName;
   
   public Name(String firstName, String lastName)
   {
      this.firstName = firstName;
      this.lastName = lastName;
   }
}

Now the two different versions of the variable can be differentiated. The firstName variable with 'this' in front is the instance field, while the naked one is still the parameter.

Shadowing is not necessarily the wrong thing to do, but you must be careful and use 'this' when appropriate.
