int Division

When dealing with ints, much of the math works the way that we expect it to. Consider the following snippet of code:

int i = 15;
System.out.println( i );
i = i + 5;
System.out.println( i );
i = i * 2;
System.out.println( i );
i = i - 3;
System.out.println( i );

Output:
15
20
40
37

These are the basic math operations and each time the variable changes it is to an expected value.

int Division Truncates

The weird stuff happens with division gets involved. The problem derives from the fact that an int divided by another int should result in a an int, but not all divisions result in a whole number. So the rule is that whenever two ints are divided, the result is the quotient. So saying 10/4 is asking "how many whole times does four go into ten?"; the answer is a flat 2, not the 2.5 that we are used to.

This is not a rounding operation, it is a truncation, meaning that the decimal is simply chopped off.

15/2 --> 7
48/9 --> 5
5/11 --> 0
18/5 --> 3
15/2 --> 7

Example

Sometimes the use of int division can be a good thing. Consider the following simple program, where the user puts in a number of days and is told how many full weeks there are.

public class NumWeeks
{
   public static void main()
   {
      String numDayStr = JOptionPane.showInputDialog("How many days?");
      int numDays = Integer.parseInt(numDayStr);
      int numWeeks = numDays/7;

      System.out.println(numDays + " days is " + numWeeks + " weeks");
   }
}

If the user enters 17 days, the int division will give the quotient of 17/7, which is 2 full weeks. The number of leftover days is not calculated in the program. They are essentially discarded when the int division occurs. If we wanted, we could do more math to calculate the remaining days. Read more about this in the Modulus Notes.

Be Careful: Order of Operations

Since int division is an operation that loses some information, the order of operations is now very important. Consider the expression below.

10 + 3 * 5 / 2 - 6 * 2

Order of operations puts multiplication, mod, and division all on the same level. java will process these from left to right. In this expression, the key is

3 * 5 / 2

If we follow proper order of operations, 3*5 comes first and the result would be 15/2, which is 7.

If we don't follow proper order of operations and we calculate the 5/2 first, the result is 3*2, which is 6.

Common Error: Casually assigning to a double

One common attempt at solving the int division issue is using the incorrect strategy below.

int x = 15;
int y = 25;
double div = x/y;

The person who programs this thinks "since div is a double, I will get non-int division and I expect 0.6". In reality there are two steps: First the x/y occurs and then result gets assigned to the div variable. Looking at x/y we can see that it is still int/int and therefore results in a int. In this case 15/25 is a 0. Only after that 0 is determined does the value get assigned to a double. At the end of the snippet above, div has a value of 0.0.

Common Error: Fractions

Often times int division can get us into trouble, especially when it comes to fractions. Consider the flawed calculation of the area of a triangle.

int base = //some number from the user
int height = //some number from the user
int area = 1/2*base*height;
System.out.println("The area of the triangle is: "+area);

No matter what the variables are, the output will always be The area of the triangle is: 0

This is because int division causes the fraction 1/2 to always be a 0. After that, the base and height are irrelevant.
